Invoice Processing

Upload invoices, extract data with AI, review and approve — with full audit trail.

What This App Does

This is an end-to-end invoice processing workflow. Users upload invoice documents (PDFs, images), and an AI automation automatically reads the document and extracts structured data — vendor name, invoice number, dates, amounts, line items, and GL codes. The extracted invoice lands in a review queue where a human can correct any mistakes and then approve or reject it.

The app also includes:

  • Vendor management with default GL codes — so the AI can suggest coding automatically
  • Line item extraction — each invoice line gets its own record with description, quantity, price, and GL code
  • Activity timeline — every status change, approval, rejection, and comment is logged per invoice
  • Full audit trail — all creates, updates, and deletes across every collection are tracked with before/after state
  • Dashboard with invoice counts by status, spend by vendor, spend by GL code, and invoice aging

Invoice Lifecycle

Every invoice moves through a defined status flow. The first two transitions happen automatically — the user only intervenes at the review stage.

draft → processing → review → approved or rejected
  1. Upload — User uploads a document via the "New Invoice" button. A record is created in ip_invoices with status: draft and the attached file.
  2. Auto-trigger — The ip_trigger_extract hook fires on after_create. It sets the status to processing and queues the extract_invoice automation.
  3. AI extraction — The automation downloads the document, calls llm.extract_text() to read it, parses vendor info, amounts, dates, and line items. It updates the invoice record, creates ip_line_items records, and suggests a GL code from the vendor's default. Status moves to review.
  4. Human review — The invoice appears in the review queue. The detail page shows the original document side-by-side with the extracted fields in an editable form. The user can correct any field the AI got wrong.
  5. Approve or reject — User clicks Approve (with an optional note) or Reject (with a required reason). A comment is recorded and the status is set accordingly.

Throughout this flow, the ip_create_status_comment hook automatically logs every status change as a system comment in ip_comments, creating a complete activity timeline for each invoice.

Automations & Hooks

extract_invoice automation

The core AI automation. Takes an invoice_id as input, downloads the attached document, calls llm.extract_text() to read it with AI vision, then:

  1. Parses vendor name, invoice number, dates, amounts from the AI response
  2. Creates ip_line_items records for each extracted line
  3. Looks up the vendor in ip_vendors and suggests their default GL code
  4. Updates the invoice record with all extracted fields
  5. Sets status to review

External ID: invoice-processing:extract_invoice

ip_trigger_extract hook

Fires on ip_invoices after_create. Sets status to processing and queues the extract_invoice automation. This is what makes extraction fully automatic — the user just uploads, and everything else happens in the background.

ip_create_status_comment hook

Fires on ip_invoices after_update when the status field changes. Creates a system comment in ip_comments recording the old and new status, so the activity timeline always reflects every transition.

Audit hooks hooks

A set of hooks (ip_audit_invoices_*, ip_audit_vendors_*, ip_audit_gl_accounts_*, ip_audit_line_items_create, ip_audit_comments_create) that fire on create/update/delete events and log the change to ip_audit_log with before/after state.

Hook ordering matters: ip_trigger_extract must fire before ip_create_status_comment on invoice creation, since the trigger hook sets the initial status change that the comment hook then logs.

Key Design Decisions

  • Upload-first creation: Invoices are created by uploading a document, not by filling out a form. The AI does the data entry.
  • Line items as separate records: Each line item is its own record in ip_line_items (not JSON inside the invoice). This allows per-line GL coding and easier querying.
  • GL suggestion from vendor default: If a vendor has a default_gl_code, the automation pre-fills it. The user can always override during review.
  • Comments as activity log: The ip_comments collection serves as both user comments and system events, unified in a single timeline per invoice.
  • All collections prefixed with ip_: This avoids name collisions if multiple templates are deployed to the same Lumera account.
